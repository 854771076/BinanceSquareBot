# 技术选型报告

## 一、项目定位与需求分析

### 1.1 项目基本信息

| 属性 | 内容 |
|------|------|
| 项目名称 | BinanceSquareBot |
| 项目类型 | AI应用 / CLI工具 |
| 项目规模 | 小型 |
| 预期开发周期 | 1周 |
| 团队规模 | 1人 |

### 1.2 核心需求概述

每小时增量爬取Fn新闻列表，使用SQLite去重已处理文章，遍历配置的多个API密钥，为每条新闻调用LLM生成符合规范的币安广场推文，并通过币安广场API发布。

核心功能：
1. 定时增量爬取Fn新闻列表
2. SQLite存储去重（基于文章URL MD5）
3. 多API密钥轮询
4. LangGraph + LangChain 构建AI工作流生成推文
5. 推文必须满足格式要求（100-800字符，≤2个话题#，≤2个代币$）
6. 通过币安广场API发布推文
7. CLI命令行工具使用Typer + Rich
8. 使用GitHub Actions做定时调度

### 1.3 关键约束条件

| 约束类型 | 约束内容 |
|----------|----------|
| 技术约束 | 用户已指定Python + LangChain + LangGraph技术栈 |
| 性能约束 | 每小时执行一次，单次处理数据量小，性能要求不高 |
| 安全约束 | API密钥需要安全存储，不能提交到代码库 |
| 部署约束 | 通过GitHub Actions部署，无需服务器 |
| 时间约束 | 快速交付 |

## 三、架构模式选择

### 3.1 架构模式对比分析

| 架构模式 | 适用场景 | 优势 | 劣势 | 本项目适配度 |
|----------|----------|------|------|--------------|
| 单体分层 | 小型工具应用 | 简单、部署容易 | 扩展性差 | ★★★★★ |
| 前后端分离 | 中型Web应用 | 清晰边界、独立开发 | 需API设计 | ★☆☆☆☆ |
| 微服务 | 大型企业系统 | 高扩展、独立部署 | 复杂、运维重 | ★☆☆☆☆ |
| Agent编排 | AI应用 | 工作流驱动、灵活 | 状态管理复杂 | ★★★★☆ |

### 3.2 架构模式选择结论

| 属性 | 内容 |
|------|------|
| 选定架构 | 单体分层 + Agent编排 |
| 选择理由 | 项目规模小，功能单一，单体分层足够简单；AI工作流部分使用LangGraph做Agent编排，满足状态管理需求 |
| 分层设计 | 1. CLI层（命令入口，参数解析） 2. Service层（爬取服务、AI生成服务、发布服务） 3. 数据层（SQLite存储） 4. Model层（数据结构定义） |
| 模块划分策略 | 按功能模块划分，清晰职责分离 |

### 4.1 代码风格规范

| 规范项 | 规范内容 |

| 配置文件路径 | pyproject.toml |
| 自动校验 | pre-commit hook |

### 4.2 命名规范

| 类别 | 规范 |
|------|------|
| 文件命名 | snake_case（如 `fn_spider.py`） |
| 变量命名 | snake_case（如 `article_url`） |
| 常量命名 | UPPER_SNAKE_CASE（如 `MAX_RETRIES`） |
| 类命名 | PascalCase（如 `SquareBot`） |
| 函数命名 | snake_case，动词开头（如 `fetch_news_list`） |

### 4.3 目录规范


### 4.4 安全规范

| 规范项 | 规范内容 |

| 传输安全 | HTTPS强制（爬取和API调用都使用HTTPS） |
